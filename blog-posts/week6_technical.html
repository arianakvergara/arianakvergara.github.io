Blocks, Procs & Lambdas

September 11, 2014

"A block is a bit of executable code." That seems to be the general concensus in regards to block on the numerous websites, blogs and resources I have read. Furthermore, it all seems to be about closure (just like in most relationships). Closure in Ruby-lingo, fills three distinct criteria: 1.) It can be passed around as a value. 2.) It can be executed on demand by anyone who has that value. 3.) It can refer to variables within the context of the variables' creation. Blocks, Procs and Lambdas all have distinct closures within Ruby. Let's go through them.

Blocks, believe it or not, are the least straight-forward of the three. Blocks are not explicitly stated: blocks are considered nameless methods. Methods like each and times are easily combined with blocks which is what makes them most convenient. In addition, they are simple.

Procs are memorable blocks. Blocks unfortunately are not complete objects. They do not have the capacity to be saved to variables and do not have all the powers or abilities to be saved to real objects. When you write a proc once, you can use it numerous times again. A block is a version of a Proc that can only be used once. Therefore, a Proc is a procedure that can be used more than once.

Proc.new defines a new procedure. We can pass Proc to a method that would otherwise normally take a block. Procs are complete objects that have all the powers and abilities of objects. Procs can be used over and over without any sort of rewrite. You do not have to retype the content of a block everytime you want to execute a certain bit of code. Procs are useful as an attribute and you can save them as a variable with the ampersand (&proc).

Now you may be wondering when one should use a block and when one use a Proc. Excellent question.

When should I use a block?

If your method is breaking an object down into smaller pieces and you want your user to interact with bits of code.

If you want to run multiple expression automatically. i.e. within a database.


When should I use a Proc?

If you want to reuse a block of code multiple times.

If your method has more than one callback.

Now for Lambdas. Lambdas are very very similar to Procs but the KEY difference is that Lambdas check the number of arguments passed. In fact, Lambdas are useful in the same situations in which you would use a Proc. However, there are two differences. Again, lambdas check the number of arguments passed to it while a proc does not. A lambda will actually throw an error if you pass it the wrong number of arguments whereas a proc will ignore unexpected arguments with a nil. In addition, procs and lambdas treat the return keyword distinctly.'Return' inside of a lambda triggers the code right outside of the lambda code. 'Return' inside of a proc triggers the code outside of the code where the proc is being executed. As Proc is to &, Lambda is to :. Therefore one can call (:lambda).

Blocks, procs or lambdas regardless for what the situation calls for they all provide closure. It is simply a matter of what kind of closure you want.
